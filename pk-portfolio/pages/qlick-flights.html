Prototype

Booking a flight is arduous. Finding a flight within your price and date range means navigating multiple sites to see which airline is offering what. The mental gymnastics to calculate the best fit for your itinerary is exhausting. Not to mention, the frustration that can occur when you finally get to the end of the process and realize your price is different than what you were first promised.


Create a place users want to be, and never give them a reason to leave. A booking experience that minimizes input, presents accurate information and feels personalized will elevate itself in the market. In this busy world, users don't have time to deal with anything less.


That was our main goal with Qlick Flights, ease. Ease of use and ease of mind. An aesthetically simple design reinforcing a sense of usability and trust. Making it the go-to suggestion when people ask “how do you book your flights?”

My Role

Using directed storytelling we studied the standard process of booking a flight online. During research, our subjects recounted one memorable booking experience. All users expressed dissatisfaction with some part of their booking experiences. Each individual reported untrustworthy pricing, predatory advertisements and promotions, and an overwhelming amount of data entry. With this our goal statement was born.

Goal Statement
My user is a fiscally-responsible person who wants to book flights that have accurate pricing. They want a streamlined experience that allows them to book with minimal input.

Pen to Paper

Qlick Flights search page.

I began sketching my ideas on paper and came up with the general flow of a first time user booking a flight. This included a home page with flight search fields, a search results page, and an itinerary page. Users were presented with minimal input on the first page. Just enough information to start searching for flights.


Next, the search results page showed the potential flights in ascending order from cheapest to most expensive. Emphasizing the important elements and reducing the required inputs means the user could select their flights, outbound and return, with just one click.

Flight search results, tickets designed to mimic layout of an airline ticket.

After selecting both flights, the user is presented with an itinerary. Outbound and return flights would be detailed along with the cost break down for each element, as a means to reinforce the user that the price they see is what they get.

Itinerary page, shows user their selected flights and the price breakdown.

When booking a flight, its unavoidable to enter the passenger information. Allowing a user to save their information for future bookings reduces the work required for future bookings. Placing this on the Itinerary page, along with the option to checkout as a guest, aims to instill trust with the user that their information is being saved for their benefit.

Sketch

Digitized wireframes, built in Sketch.

Creating the main user-arc was not enough for the prototype as users will need to interact with more pages. A wireflow provided more insight into the necessary pages.

Wireflow of required Qlick Flights pages.

For a complete booking experience we added several pages. If a user created an account they would need a way to access that account and change their details.

Account page for a Qlick Flights user.

If a user did not have an account yet they were greeted with an option to join from the home page. A three section sign-up page walks the user through the creation of a profile. Users input their personal information and preferences on how they like to fly before saving their account to an email.

Sign-up pages for a new user.

This sign up page offers a way to enhance the user experience with future bookings. Each user can select preferred airlines, what type of baggage they normally fly with, and a seat preference option. All of these elements automatically apply to future bookings saving the user time and creating a more personalized experience.


Users who had already created an account, or logged in, would view a refined search page. A reduced input search page removes the options for the preferences the user set in their profile.

Search page for a logged in user.

The user is presented with search results reflecting their preferences. If a preferred airline was set, the user receives preferred airline flight options at the top of the page.


The rest of the experience remains similar apart from the final itinerary page. This page does not require any input besides payment option. Booking a flight, once logged in, takes even less time and removes more work from the user.

Enhanced itinerary page that shows automatically completed passenger information.

Prototype

Try Prototype

I used Axure RP to create a prototype to conduct usability-tests with 4 participants. This provided invaluable insight into the way users stated they like to work with a website versus how they actually interacted. The minimal input and rapid booking process was well received by all participants. Although, several users expressed concerns that they weren’t getting all the information.


Sign up engagement proved difficult as users did not want to have to complete a registration. Once the users were prompted to create an account and try booking again, there was an immediate gain perceived. Each individual commented on how their information was stored and they did not have to do as much, creating a more positive user experience. This revealed an opportunity to onboard users with their first flight booking as opposed to having them sign up before booking.

Key Findings

In my initial research users reported a disdain for the overwhelming amount of options and screens required to book a single flight. The initial designs and prototypes proved that users appreciated a streamlined process, but questioned lack of choice. A fine balance must be struck between intrusive and inclusive. A user must be able to make all the choices they want, but report a more preferable experience when it is an opt-in instead of an opt-out. The flexibility of certain flight options could be increased in future prototypes to see if users report more confidence in their control. In future iterations it will be important to identify how much decision making can be provided to the customer before the usability suffers.
